## 2. 12家公司方案阅读记录（提炼）

### 2.1 Microsoft AIOpsLab
- 方案定位：面向AIOps Agent的设计-开发-评估一体化实验框架。
- 关键能力：故障注入、负载生成、标准化接口（ACI）、可重复评测。
- 对本项目启发：优先建立本地benchmark harness与可回放评测流水线。

### 2.2 Ask Red Hat
- 方案定位：面向企业知识库支持场景的RAG助手。
- 关键能力：跨知识源检索、门禁与安全检查、多语言支持、AI系统边界说明。
- 对本项目启发：补齐“知识检索Agent + 安全治理卡（System Card）”。

### 2.3 Gemini Cloud Assist
- 方案定位：云资源全生命周期助手（设计/部署/运维/优化）。
- 关键能力：调查工作流（investigation）、多域上下文融合、动作建议输出。
- 对本项目启发：增加“调查模板+会话脚手架”与单页调查闭环。

### 2.4 IBM Instana
- 方案定位：全栈可观测+自动RCA平台。
- 关键能力：动态依赖图、跨层关联分析、自动事件关联与根因定位。
- 对本项目启发：增强“拓扑与证据链图谱”并做自动调查并行收集。

### 2.5 Dynatrace Davis AI
- 方案定位：预测AI + 因果AI + 生成式AI融合。
- 关键能力：相关性到因果性提升、风险排序、自然语言操作入口。
- 对本项目启发：引入因果评分层与根因候选排序机制。

### 2.6 TrueFoundry
- 方案定位：企业级AI平台与网关治理。
- 关键能力：统一模型入口、路由与回退、成本与配额治理、审计。
- 对本项目启发：建设本地“调用审计+成本估算+配额控制”面板。

### 2.7 DeepFlow
- 方案定位：高性能可观测驱动的智能体运维。
- 关键能力：全栈信号感知、分钟级诊断、知识图谱与模式库。
- 对本项目启发：工具层预留eBPF/trace接入规范，强化实时关联。

### 2.8 Microsoft Argos
- 方案定位：LLM驱动、可解释异常检测规则生成。
- 关键能力：规则生成-修复-审查闭环，多Agent迭代优化。
- 对本项目启发：新增“规则建议Agent”，输出可执行规则草案。

### 2.9 OpenDerisk
- 方案定位：工业级SRE多Agent框架。
- 关键能力：SRE分工协作、推理引擎、知识引擎、MCP协议、HITL。
- 对本项目启发：完善“Agent消息协议+证据链+HITL反馈”。

### 2.10 STRATUS
- 方案定位：自治可靠性工程多Agent系统。
- 关键能力：状态机控制、事务无退化（TNR）、失败回滚。
- 对本项目启发：引入修复动作状态机与No-Regression Gate。

### 2.11 AWS DevOps Agent
- 方案定位：云运维自治助手与平台集成。
- 关键能力：运维工具链深度集成、双控制台、调查与执行协同。
- 对本项目启发：抽象连接器协议，统一监控/工单/代码上下文。

### 2.12 RCAgent
- 方案定位：工具增强型LLM自主RCA框架。
- 关键能力：ReAct循环、控制器+专家、观察压缩（OBSK）、结构化输出。
- 对本项目启发：做“多轨迹自一致性+上下文压缩+强结构化输出”。

## 3. 横向对比（对本项目最相关维度）

| 维度 | 业界共性最佳实践 | 当前项目现状 | 差距 |
|---|---|---|---|
| 评测体系 | 标准故障集 + 自动评分 + 可复现 | 有样例，缺统一评分基线 | 高 |
| 证据体系 | Claim-Evidence-Tool可追溯链路 | 有过程记录，证据对象不统一 | 中高 |
| 因果推理 | 因果与相关分层，输出置信度 | 以文本结论为主 | 高 |
| 工具生态 | MCP/连接器可插拔 + 治理策略 | 已有3类工具，扩展协议弱 | 中高 |
| 自治修复 | 状态机门禁 + 回滚保障 | 以分析建议为主，执行门禁不足 | 高 |
| 治理与合规 | 审计、成本、配额、系统边界卡 | 有日志，缺完整治理面板 | 中高 |
| 产品体验 | 调查工作台、时间线、图谱化证据 | 主流程可用，分析工作台不足 | 中 |

## 4. 对本项目的优化改进计划（落地版）

说明：以下计划在“暂不引入外部数据库”的约束下设计，采用本地文件/内存存储。

## P0（1周）：评测与可观测基础
1. 建立 `benchmark` 目录与标准故障集（日志、症状、预期根因）。
2. 新增基线评分脚本：Top1/Top3、平均耗时、失败率、工具调用成功率。
3. 每次会话产出执行谱系（输入摘要、Agent结论、工具I/O、耗时、置信度）。
4. 增加会话回放器（按时间线重放关键事件）。

验收：
- 一条命令可完成基准测试并生成报告；
- 任意失败会话可回放。

## P1（1-2周）：推理质量升级
1. 引入统一证据对象模型：`Evidence/Claim/Hypothesis`。
2. 引入因果评分：相关证据与因果证据分层打分。
3. 对关键Agent启用低成本多轨迹自一致性投票。
4. 新增“规则建议Agent”（异常阈值/窗口/触发条件草案）。

验收：
- Top1根因命中率提升；
- “需要进一步分析”结论占比显著下降。

## P2（1-2周）：工具生态标准化
1. 实现本地工具注册中心：schema、权限、超时、审计级别。
2. 抽象连接器接口：`RepoConnector/TelemetryConnector/AssetConnector/TicketConnector`。
3. 工具调用安全网：白名单、参数脱敏、输出截断、失败降级。
4. 前端展示工具健康状态、真实调用输入摘要与返回摘要。

验收：
- 新增工具无需改核心编排；
- 工具调用可审计、可追踪。

## P3（1-2周）：可控自治修复
1. 引入修复状态机：`PROPOSED -> SIMULATED -> APPROVED -> EXECUTED -> VERIFIED`。
2. 增加No-Regression Gate：关键SLO前后对比，不达标阻断执行。
3. 输出回滚建议与风险等级，默认人工确认。

验收：
- 每个修复动作有风险与回退方案；
- 高风险默认不可自动执行。

## P4（2周）：产品化体验闭环
1. 新增“调查工作台”：时间线、证据链图谱、决策摘要。
2. 新增“评测中心”：版本对比、准确率趋势、耗时趋势。
3. 新增“治理中心”：token/成本估算、配额、审计检索。
4. 反馈学习闭环：结论采纳/驳回/修订记录驱动提示词与规则优化。

验收：
- 用户可在单页完成调查到结论确认；
- 连续两周质量指标可量化提升。

## 5. 本项目当前优先改进点（建议先做）

1. 先做P0 + P1：先把“质量可衡量、结论可解释”做扎实。
2. 再做P2：把工具生态做成标准接口，降低后续扩展成本。
3. 最后做P3 + P4：在可控前提下推进自治修复与产品化闭环。
