## 1. 背景与目标

当前项目已具备：
- LangGraph 多 Agent 协作（主 Agent + 专家 Agent）
- 责任田映射、工具调用、事件流、报告生成
- 前后端闭环与基础可观测日志

下一阶段目标：
- 从“可用”升级到“可评估、可治理、可扩展、可自治”
- 对齐业界 12 个方案中的高价值能力，形成可执行路线图

范围约束：
- 暂不引入外部数据库（继续使用本地文件/内存存储）
- 保持现有前端交互主流程不变，在此基础上增强

## 2. 12 个方案 -> 本项目可借鉴能力映射

1. AIOpsLab（Microsoft）
- 借鉴点：标准化评测基线、故障注入、可重复实验
- 本项目落地：建设本地 `benchmark harness`，对每次版本做自动评分（RCA 准确率/耗时/稳定性）

2. Ask Red Hat
- 借鉴点：面向知识库问答、明确安全边界、系统卡（AI System Card）
- 本项目落地：新增“运维知识检索 Agent”+ 输出安全边界声明 + 模型/数据来源披露

3. Gemini Cloud Assist
- 借鉴点：从设计到运维的全生命周期助手、调查模式（investigations）
- 本项目落地：新增“调查会话模板”（从告警到定位到建议），提供标准调查脚手架

4. IBM Instana
- 借鉴点：一键自动调查、跨层依赖关联、证据自动收集
- 本项目落地：实现“自动调查模式”按钮，按拓扑依赖并行收集证据

5. Davis AI（Dynatrace）
- 借鉴点：预测 + 因果 + 生成式（hypermodal）融合
- 本项目落地：在裁决层引入“因果证据权重评分 + 预测风险评分”

6. TrueFoundry
- 借鉴点：LLMOps 治理（RBAC、审计、成本、配额）
- 本项目落地：增加本地治理面板（调用配额、成本估算、审计追踪）

7. DeepFlow
- 借鉴点：eBPF 零侵入观测、全栈信号关联
- 本项目落地：工具层预留 eBPF/trace 接入规范（先接口适配，后数据接入）

8. Argos（Microsoft）
- 借鉴点：可解释异常规则自动生成
- 本项目落地：新增“规则建议 Agent”，输出可回放规则（阈值/窗口/条件）

9. OpenDerisk
- 借鉴点：多 Agent + 可视化证据链 + MCP 工具生态
- 本项目落地：证据链图谱化（Claim-Evidence-Tool-Conclusion）+ MCP 工具注册目录

10. STRATUS
- 借鉴点：状态机+安全规约（No-Regression）控制自治修复
- 本项目落地：引入“修复动作门禁状态机”（模拟执行 -> 人工确认 -> 自动执行）

11. AWS DevOps Agent
- 借鉴点：与监控/CI/CD/工单/协作平台深度集成，MCP 扩展
- 本项目落地：标准化连接器协议（Webhook + MCP），统一事件/变更上下文

12. RCAgent
- 借鉴点：工具增强、轨迹自一致性、上下文稳定策略
- 本项目落地：多次推理轨迹投票（self-consistency）+ 长上下文裁剪/记忆策略

## 3. 改进路线（P0-P4）

## P0（稳定性与可观测性基线，1 周）

目标：先解决“看不清、评不了、不可复现”的问题。

任务：
1. 增加端到端评测基线脚本（本地）
- 输入：标准故障样例集（日志+症状+预期根因）
- 输出：准确率、Top-K 命中率、平均耗时、失败率

2. 增强运行时审计
- 每个 Agent 记录：输入摘要、工具调用、结论、置信度、耗时
- 每次会话输出“执行谱系 JSON”（可复盘）

3. 故障注入样例集固化
- 新增 `benchmarks/incidents/*.md|json`
- 支持 smoke + benchmark 两套运行模式

验收指标：
- 单次 benchmark 可自动完成并产出评分报告
- 失败会话可 100% 回放关键过程

---

## P1（多 Agent 推理质量提升，1-2 周）

目标：从“串行发言”升级为“证据驱动辩论”。

任务：
1. 引入“证据对象模型”
- `Evidence(id, source, tool, snippet, confidence, timestamp)`
- 每个结论必须引用至少 1 条证据（可追溯）

2. 引入轨迹自一致性（RCAgent 思路）
- 对关键 Agent（Log/Code/Judge）做 N 次低成本推理
- 使用投票/加权合并结论

3. 因果评分层（Davis/Instana 思路）
- 区分“相关证据”与“因果证据”
- 输出最终“因果置信度”

4. 异常规则建议（Argos 思路）
- 从会话中自动提取可执行规则建议（阈值、窗口、触发条件）

验收指标：
- 根因 Top1 准确率提升 >= 15%
- “需要进一步分析”占比下降 >= 40%

---

## P2（工具生态与连接器标准化，1-2 周）

目标：从“3 个定制工具”升级为“可扩展工具生态”。

任务：
1. MCP 工具注册中心（本地）
- 工具元数据：名称、输入 schema、超时、权限、审计级别
- 前端展示工具注册状态与健康度

2. 连接器统一协议
- 统一 `TelemetryConnector / RepoConnector / TicketConnector`
- 当前内置：本地日志、Git、责任田文件
- 预留：Datadog/Splunk/GitLab/Jira/ServiceNow

3. 工具调用安全网
- 命令白名单、路径白名单、参数脱敏、输出截断
- 工具失败自动降级与补偿策略

验收指标：
- 新增一个工具不改动核心编排（插件式接入）
- 所有工具调用都有可审计 I/O 记录

---

## P3（自治修复与安全治理，1-2 周）

目标：从“分析建议”升级到“可控执行”。

任务：
1. 修复动作状态机（STRATUS 思路）
- `PROPOSED -> SIMULATED -> APPROVED -> EXECUTED -> VERIFIED`
- 默认仅到 `APPROVED`（人工确认）

2. 回归风险保护（No-Regression Gate）
- 执行动作前后自动对比关键 SLO/错误率
- 未达标自动回滚策略建议

3. AI System Card（Ask Red Hat 思路）
- 模型来源、适用边界、已知风险、禁用场景、审计方法

验收指标：
- 每条修复建议都有风险等级与回退方案
- 高风险动作默认禁止自动执行

---

## P4（产品化体验与持续学习，2 周）

目标：建立“持续变好”的产品闭环。

任务：
1. 调查工作台（Gemini/Instana 思路）
- 一键调查模板（告警 -> 证据 -> 根因 -> 建议）
- 时间线 + 证据链图谱 + 决策摘要

2. 用户反馈学习闭环
- 对每次结论进行“采纳/驳回/修订”反馈
- 自动形成规则优化与提示词优化候选

3. 成本与性能面板（TrueFoundry 思路）
- 每会话 token、耗时、工具成本估算
- 慢节点诊断与优化建议

验收指标：
- 用户可在单页完成“调查-复核-结论确认”
- 回归版本质量可量化提升（连续 2 周）

## 5. 里程碑与优先级

- M1（P0 完成）：可评估、可回放、可审计
- M2（P1 完成）：结论质量显著提升，减少空洞结论
- M3（P2 完成）：工具生态可扩展、可治理
- M4（P3 完成）：可控自治修复闭环
- M5（P4 完成）：产品化体验 + 持续学习

优先顺序建议：
1. 先做 P0 + P1（质量与可信度）
2. 再做 P2（扩展性）
3. 最后做 P3 + P4（自治与产品化）

---

## 6. 风险与对策

风险：
1. 多 Agent 成本和时延上升
2. 证据链复杂度增加导致维护成本上升
3. 工具接入扩展后安全风险扩大

对策：
1. 分层推理（轻量预筛 + 深度分析）
2. 强制证据 schema 与字段校验
3. 工具权限最小化 + 全量审计 + 高危动作门禁

---

## 7. 成功定义（Done）

当满足以下条件，视为“对齐 12 方案经验的第一阶段完成”：
1. 基准集 RCA Top1 准确率与稳定性达到目标阈值
2. 每个结论具备可视化证据链与可追溯来源
3. 工具接入具备标准协议与治理能力
4. 修复动作具备安全门禁与回归保护
5. 前端提供一体化调查体验与复盘能力
